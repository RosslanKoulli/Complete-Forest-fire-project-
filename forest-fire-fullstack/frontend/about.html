About this project

Final-year individual project for CI601 at the University of Brighton.

The problem

Wildfires destroy ecosystems, displace communities, and contribute meaningfully to greenhouse gas emissions. Predicting where and when fires are likely to ignite would let fire services pre-position resources and trigger preventative measures. The task here is the simpler of two related problems: given the current weather and fuel state at a location, estimate the probability of ignition. The harder problem, predicting exact timing and location of an unstarted fire requires data sources well beyond an undergraduate project.

Models compared

Random Forest
Bagging ensemble of decision trees

Strong baseline for tabular data with mixed feature types. Built-in feature importance. Resistant to overfitting on small datasets.

XGBoost
Gradient boosting

State-of-the-art for tabular classification. Each tree corrects errors of the previous. Generally outperforms Random Forest on the Algerian dataset in published benchmarks.

MLP Neural Network
Multilayer perceptron

Tests whether neural networks can match tree ensembles on small tabular data. Spoiler: usually no, and that null result is itself a finding.

Dataset

Two regional datasets combined into one training corpus of 761 samples:

  • UCI Forest Fires (Portugal, Montesinho Natural Park) -- 517 records with eleven features including FWI components.
  • Algerian Forest Fires (Bejaia and Sidi Bel-Abbes) -- 244 records with the same feature set.

Both regions have Mediterranean climates -- hot dry summers, mild wet winters. The combined dataset captures fire-weather relationships that are specific to this climate zone.

Why a separate API and frontend?

The original prototype used Streamlit, which couples model serving and UI into a single Python process. That choice was timeline-driven. This version separates them -- a FastAPI backend serves predictions and simulations as REST and WebSocket endpoints, and the frontend is a static set of HTML, CSS, and JavaScript files served by the same FastAPI instance. The separation is what production ML systems do, and it lets the two layers evolve independently. The auto-generated API docs at /docs are a real artefact: anyone can call the API themselves with no other tooling.

The frontend is hand-written vanilla JavaScript -- no React, no build step, no npm. Every line is direct DOM manipulation and fetch calls. Trade-off: more code volume than a framework version, but every line is readable and auditable, and the page ships zero JavaScript framework bytes.

Map-based interaction

The Map page is the primary interactive surface. It uses Leaflet for the base map, the H3 hexagonal hierarchical indexing system for tessellation, and Open-Meteo for live weather. The flow is: draw a rectangle, get hexagons coloured by fire probability, click any hexagon to ignite a fire, watch the spread animate across the surrounding hexes.

Each hex's prediction is built from real conditions. The backend fetches the last seven days of daily weather for the hex's centroid, walks them through the Van Wagner FWI System equations to compute FFMC, DMC, DC, and ISI, then runs all three trained models on the resulting feature vector. Weather lookups are cached at 0.1-degree resolution (about 11 km) so a 200-hex selection over a small region produces a handful of API calls rather than two hundred.

The fire spread itself is a hex-based cellular automaton. Each burning hex has a probability of igniting each of its six neighbours per time step, weighted by wind direction (against the wind: low; with the wind: high), vegetation moisture, and the neighbour's underlying fire probability from the ML models. The user-controllable parameters in the side panel directly drive the simulation. State streams over WebSocket frame by frame, rendered as colour changes on the existing hex polygons rather than redrawing the whole map each frame.

Honest scoping

Mediterranean climate only. The models learned the relationship between fire-weather conditions and ignition from two regional datasets. They will generalise reasonably to other Mediterranean basins but should not be trusted on tropical, boreal, grassland, or oceanic climates. Production global fire prediction needs region-specific models, satellite-derived vegetation, terrain, and orders of magnitude more training data -- a multi-year research effort.
